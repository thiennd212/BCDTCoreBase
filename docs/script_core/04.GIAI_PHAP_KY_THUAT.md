### 1.4. RLS & Session Context

Để Row-Level Security (RLS) áp dụng đúng (user chỉ thấy data đơn vị mình — 01 ORG-02, ORG-04), mọi request sau khi xác thực **phải** set session context trên connection dùng cho truy vấn.

**Vị trí thực hiện:** Middleware trong BCDT.Api, chạy **sau** `UseAuthentication()`, **trước** `UseAuthorization()` và endpoint.

**Logic (đặc tả):**

1. Lấy **UserId** từ Claims (vd `ClaimTypes.NameIdentifier` hoặc custom "UserId") sau khi JWT được validate.
2. (Tùy chọn) Lấy **OrganizationId** hoặc danh sách org từ Claims / service nếu RLS predicate cần.
3. Trên **connection** mà request dùng cho DB (EF/Dapper): gọi `EXEC sp_set_session_context N'UserId', @UserId` (và tương tự cho key khác nếu dùng).
4. Đảm bảo mọi truy vấn trong request dùng **cùng connection** (hoặc cùng scope) đã set session context.

**Lưu ý:** Session context gắn với session (connection); connection phải mở trước khi gọi `sp_set_session_context`. Có thể gói trong scope per-request. **Không** bypass RLS.

### 1.5. SaveOrchestrator: Thứ tự ghi và transaction

Khi ghi dữ liệu từ Save Orchestrator vào 3 layer, **thứ tự** và **transaction** phải đảm bảo tính nhất quán.

**Thứ tự ghi (bắt buộc):**

1. **Layer 2 – DataRow:** Ghi vào `BCDT_ReportDataRow` (dữ liệu có thể query).
2. **Layer 2.5 – Summary:** Ghi vào `BCDT_ReportSummary` (tổng pre-calc).
3. **Layer 1 – Presentation:** Ghi vào `BCDT_ReportPresentation` (WorkbookJson).

**Transaction:** Toàn bộ 3 bước ghi trong **một transaction** (vd `DbContext.Database.BeginTransaction()` hoặc `TransactionScope`). Nếu **bất kỳ bước nào** lỗi → **rollback hết**, không lưu nửa chừng. Commit chỉ khi cả 3 bước thành công.

## 2. Data Binding Engine

### 2.1. Kiến trúc

```csharp
public interface IDataBindingEngine
{
    Task<object> ResolveAsync(DataBindingConfig config, BindingContext context);
}

public class DataBindingEngine : IDataBindingEngine
{
    private readonly Dictionary<string, IBindingResolver> _resolvers;
    
    public DataBindingEngine()
    {
        _resolvers = new Dictionary<string, IBindingResolver>
        {
            ["Static"] = new StaticBindingResolver(),
            ["Database"] = new DatabaseBindingResolver(),
            ["API"] = new ApiBindingResolver(),
            ["Formula"] = new FormulaBindingResolver(),
            ["Reference"] = new ReferenceBindingResolver(),
            ["Organization"] = new OrganizationBindingResolver(),
            ["System"] = new SystemBindingResolver()
        };
    }
    
    public async Task<object> ResolveAsync(DataBindingConfig config, BindingContext context)
    {
        var resolver = _resolvers[config.BindingType];
        var value = await resolver.ResolveAsync(config, context);
        
        // Apply transformation if specified
        if (!string.IsNullOrEmpty(config.TransformExpression))
        {
            value = await TransformAsync(value, config.TransformExpression);
        }
        
        // Cache if configured
        if (config.CacheMinutes > 0)
        {
            await _cache.SetAsync(config.CacheKey, value, TimeSpan.FromMinutes(config.CacheMinutes));
        }
        
        return value;
    }
}
```

### 2.2. Binding Types

| Type | Use Case | Example |
|------|----------|---------|
| **Static** | Fixed values | Year = "2026" |
| **Database** | DB lookup | Province name from Organization |
| **API** | External data | Exchange rate from API |
| **Formula** | Calculations | =SUM(B2:B10) |
| **Reference** | Entity reference | Project.Name, Project.Value |
| **Organization** | Current org info | CurrentOrg.Name, CurrentOrg.Code |
| **System** | System values | CurrentDate, CurrentUser.Name |

### 3.2. Implementation

```csharp
public class ExcelGeneratorService : IExcelGeneratorService
{
    public async Task<WorkbookJson> GenerateAsync(
        int formDefinitionId,
        int organizationId,
        int reportingPeriodId)
    {
        // 1. Load form definition with all related data
        var form = await _formRepository.GetWithDetailsAsync(formDefinitionId);
        
        // 2. Create workbook from template or empty
        using var workbook = form.TemplateFile != null
            ? _spreadsheet.LoadDocument(form.TemplateFile)
            : _spreadsheet.CreateNewDocument();
        
        // 3. Build binding context
        var context = new BindingContext
        {
            OrganizationId = organizationId,
            ReportingPeriodId = reportingPeriodId,
            CurrentUser = _currentUser
        };
        
        // 4. Process each sheet
        foreach (var sheet in form.Sheets.OrderBy(s => s.SheetIndex))
        {
            var worksheet = workbook.Worksheets[sheet.SheetIndex];
            
            // 5. Resolve data bindings
            await ResolveDataBindingsAsync(worksheet, sheet.Columns, context);
            
            // 6. Generate dynamic rows
            await GenerateDynamicRowsAsync(worksheet, sheet.Rows, context);
            
            // 7. Apply formatting
            ApplyFormatting(worksheet, sheet.Cells);
            
            // 8. Apply cell protection
            ApplyProtection(worksheet, sheet.Cells);
        }
        
        // 9. Convert to JSON for frontend
        return _spreadsheet.SaveToJson(workbook);
    }
}
```

### 4.2. Implementation

```csharp
public class WorkflowEngine : IWorkflowEngine
{
    public async Task<WorkflowResult> ProcessActionAsync(
        int instanceId,
        WorkflowAction action,
        int approverId,
        string comments)
    {
        var instance = await _repository.GetInstanceAsync(instanceId);
        var currentStep = await _repository.GetStepAsync(
            instance.WorkflowDefinitionId, instance.CurrentStep);
        
        // Validate approver has permission
        if (!await CanApproveAsync(approverId, currentStep))
        {
            return WorkflowResult.Unauthorized();
        }
        
        // Record approval
        await _repository.AddApprovalAsync(new WorkflowApproval
        {
            WorkflowInstanceId = instanceId,
            StepOrder = instance.CurrentStep,
            Action = action,
            ApproverId = approverId,
            Comments = comments
        });
        
        // Process action
        return action switch
        {
            WorkflowAction.Approve => await HandleApproveAsync(instance, currentStep),
            WorkflowAction.Reject => await HandleRejectAsync(instance),
            WorkflowAction.RequestRevision => await HandleRevisionAsync(instance),
            _ => throw new InvalidOperationException()
        };
    }
    
    private async Task<WorkflowResult> HandleApproveAsync(
        WorkflowInstance instance, WorkflowStep currentStep)
    {
        var totalSteps = await _repository.GetTotalStepsAsync(instance.WorkflowDefinitionId);
        
        if (instance.CurrentStep >= totalSteps)
        {
            // Final approval
            instance.Status = WorkflowStatus.Approved;
            instance.CompletedAt = DateTime.UtcNow;
            
            // Update submission status
            await _submissionService.UpdateStatusAsync(
                instance.SubmissionId, SubmissionStatus.Approved);
        }
        else
        {
            // Move to next step
            instance.CurrentStep++;
            
            // Notify next approvers
            await _notificationService.NotifyApproversAsync(
                instance.SubmissionId, instance.CurrentStep);
        }
        
        await _repository.UpdateInstanceAsync(instance);
        return WorkflowResult.Success();
    }
}
```

## 6. Concurrency Control

### 6.1. Optimistic Locking

```csharp
// Entity with version
public class ReportSubmission
{
    public long Id { get; set; }
    [ConcurrencyCheck]
    public int Version { get; set; }
    // ... other properties
}

// Update with version check
public async Task<bool> UpdateAsync(ReportSubmission submission)
{
    var sql = @"
        UPDATE BCDT_ReportSubmission 
        SET Status = @Status, 
            UpdatedAt = @UpdatedAt, 
            Version = Version + 1
        WHERE Id = @Id AND Version = @Version";
    
    var affected = await _db.ExecuteAsync(sql, submission);
    
    if (affected == 0)
    {
        throw new ConcurrencyException(
            "The submission has been modified by another user.");
    }
    
    return true;
}
```

### 6.2. Document Locking

```csharp
// Acquire lock when editing
public async Task<LockResult> AcquireLockAsync(long submissionId, int userId)
{
    var submission = await _repository.GetAsync(submissionId);
    
    if (submission.IsLocked && submission.LockExpiresAt > DateTime.UtcNow)
    {
        if (submission.LockedBy != userId)
        {
            return LockResult.LockedByOther(submission.LockedBy);
        }
        // Extend lock
    }
    
    submission.IsLocked = true;
    submission.LockedBy = userId;
    submission.LockedAt = DateTime.UtcNow;
    submission.LockExpiresAt = DateTime.UtcNow.AddMinutes(15);
    
    await _repository.UpdateAsync(submission);
    return LockResult.Success();
}
```

## 7. API Response & Error Format

API BCDT dùng chuẩn response thống nhất để frontend và tích hợp bên thứ ba xử lý nhất quán.

### 7.1. Success Response

- **GET (single):** `{ "success": true, "data": { ... } }`
- **GET (list, có phân trang):** `{ "success": true, "data": [ ... ], "meta": { "page": 1, "pageSize": 20, "total": 100 } }`
- **POST/PUT/DELETE:** `{ "success": true, "data": { "id": 123, ... } }` (hoặc `data: null` nếu không trả body)

| Field | Mô tả |
|-------|--------|
| `success` | Luôn `true` khi HTTP 2xx |
| `data` | Payload (object hoặc array) |
| `meta` | Chỉ có khi list có phân trang: `page`, `pageSize`, `total` |

### 7.2. Error Response

- **Format:** `{ "success": false, "errors": [ { "code": "...", "message": "...", "field": "..." } ] }`
- **HTTP status:** 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 500 Internal Server Error

| Field | Mô tả |
|-------|--------|
| `success` | Luôn `false` |
| `errors` | Mảng lỗi; mỗi phần tử có `code`, `message`; `field` (tùy chọn) cho validation |

### 7.3. Mã lỗi chuẩn (Error Codes)

| Code | HTTP | Mô tả |
|------|------|--------|
| `VALIDATION_FAILED` | 400 | Dữ liệu không hợp lệ (FluentValidation) |
| `UNAUTHORIZED` | 401 | Chưa đăng nhập hoặc token hết hạn |
| `FORBIDDEN` | 403 | Không có quyền truy cập (RBAC/RLS) |
| `NOT_FOUND` | 404 | Resource không tồn tại |
| `CONFLICT` | 409 | Xung đột (vd: lock, trùng mã) |
| `UNPROCESSABLE_ENTITY` | 422 | Logic nghiệp vụ không chấp nhận (vd: Result.Fail) |
| `INTERNAL_ERROR` | 500 | Lỗi server (không lộ chi tiết ra ngoài) |

### 7.4. Ví dụ

**Success (list có phân trang):**
```json
{
  "success": true,
  "data": [
    { "id": 1, "code": "BC_01", "name": "Báo cáo tháng 1" }
  ],
  "meta": { "page": 1, "pageSize": 20, "total": 45 }
}
```

**Error (validation):**
```json
{
  "success": false,
  "errors": [
    { "code": "VALIDATION_FAILED", "message": "Code đã tồn tại.", "field": "Code" },
    { "code": "VALIDATION_FAILED", "message": "Name không được để trống.", "field": "Name" }
  ]
}
```

**Error (business rule):**
```json
{
  "success": false,
  "errors": [
    { "code": "UNPROCESSABLE_ENTITY", "message": "Báo cáo đã bị khóa bởi user khác.", "field": null }
  ]
}
```

## 8. Security Checklist

Khi viết hoặc review code API và truy cập dữ liệu, kiểm tra các mục sau.

| # | Mục | Ghi chú |
|---|-----|--------|
| 1 | **Không hardcode secret** | Connection string, JWT secret, API key đặt trong config (vd `appsettings.Development.json`), không commit. Dùng User Secrets / env trong dev. |
| 2 | **Parameterized query** | Mọi truy vấn SQL (EF, Dapper, raw) dùng parameter, không nối chuỗi input vào câu lệnh. |
| 3 | **Validate input** | Validate đầu vào (FluentValidation, model binding). Giới hạn kích thước payload. |
| 4 | **RLS không bypass** | Mọi truy vấn dữ liệu theo đơn vị phải chạy trong session đã set `UserId`/`OrganizationId` (session context). Không tắt RLS hoặc dùng quyền elevated để bypass. |
| 5 | **JWT & HTTPS** | API production dùng HTTPS. JWT: expiry hợp lý, secret đủ mạnh, không đưa thông tin nhạy cảm vào payload. |
| 6 | **Rate limiting & caching** | Áp rate limiting cho API public; cache cẩn thận, không cache dữ liệu theo user nhạy cảm vào endpoint chung. |
| 7 | **Log không lộ thông tin nhạy** | Không log password, token, PII đầy đủ. Log lỗi với ID/context, không log stack trace ra response. |
| 8 | **Phân quyền endpoint** | Mỗi endpoint kiểm tra quyền (role/permission); trả 403 khi không đủ quyền. |
